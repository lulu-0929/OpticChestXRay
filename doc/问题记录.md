## 2026-07-08

### 问题 1：验证集过拟合——验证集97% vs 测试集80%（V1-V5）

**现象**：V1-V5 训练中验证集准确率高达 97.70%，但测试集仅 77-80%，NORMAL 召回率最低仅 48.3%。

**各版本表现**：
| 版本 | 改进 | 验证集 | 测试集 | 说明 |
|------|------|--------|--------|------|
| V1 | 初始基线 | 97.70% | 77.40% | 基础光计算框架搭建 |
| V2 | AdamW + CosineAnnealingLR | ~97% | ~77% | 优化器调整 |
| V3 | 增加数据增强 | ~97% | ~78% | 水平翻转+仿射变换 |
| V4 | Dropout=0.3 + WD | ~97% | ~78% | 正则化探索 |
| V5 | WeightedRandomSampler | 97.57% | 79.33% | 类别均衡 |

**根因分析**：
1. **验证集划分偏差**：`random_split` 随机划分，NORMAL 样本少，小验证集（仅521张）内 NORMAL 占比过低 → 验证集不能真实反映模型泛化能力
2. **FC 层过拟合**：8192→256→2 的 FC 层参数量大，模型在大量 PNEUMONIA 样本上"记住"特征
3. **NORMAL 样本不足**：NORMAL=1116 : PNEUMONIA=3318（V5训练），严重不平衡，模型偏向预测肺炎
4. **V1-V4 数据划分问题**：从训练集切 10% 做验证集，但未记录类别细分

**解决方案**：
1. V6：FC 分解为 8192→512→128→2，增加 Dropout=0.5 和标签平滑
2. V7：Focal Loss + 验证集分层采样 + NORMAL 过采样 3x

### 问题 2：Streamlit Demo 在 7860 端口运行但无法访问

**现象**：容器内 Streamlit 成功启动在 0.0.0.0:7860，但宿主机无法通过 `localhost:7860` 访问。

**原因**：
- 容器创建时未添加 `-p 7860:7860` 端口映射参数
- 容器 IP 为内网地址（172.17.0.x），不可直接访问

**解决方案**：
- 方案 A：重新创建容器带端口映射 `docker run -p 7860:7860 ...`
- 方案 B（当前采用）：改用 Gradio Demo，通过 `share=True` 生成公网可访问链接

**验证**：将 `app_demo.py` 中 `share=False` 改为 `share=True`，启动后输出 gradio.live 公网链接。

---

### 问题 3：Gradio 4.44.1 版本兼容问题

**现象**：启动 Demo 时出现 `TypeError: argument of type 'bool' is not iterable` 错误。

**原因**：Gradio 4.44.1 的 gradio_client/utils.py 内部代码兼容性问题（`if "const" in schema` 异常）。

**解决方案**：
- 设置 `analytics_enabled=False` 减少内部 API 调用
- 设置环境变量 `GRADIO_ANALYTICS_ENABLED=False`
- 如果问题持续：降级 Gradio 至 4.x 更低版本

## 2026-07-09

### 问题 4：V6 测试集 Acc=83.33%，未达 85% 门槛

**现象**：V6 训练 40 epochs，验证集最佳 98.47%，但测试集仅 83.33%。
- NORMAL 召回率：58.1%（136/234）
- PNEUMONIA 召回率：98.5%（384/390）

**根因分析**：
1. **NORMAL 样本极度不足**：训练集中 NORMAL 仅 1116 张，PNEUMONIA 有 3318 张
2. **过采样不够**：V6 用了 WeightedRandomSampler，但 NORMAL 每个 epoch 见的次数仍然有限
3. **标签平滑可能过于激进**：smoothing=0.1 对少数类惩罚较大
4. **FC 层参数量过大（4.36M）**：模型容量远超数据量，容易记住训练集

**V7 修复方案**：
- Focal Loss (γ=2.0) → 强制关注难分样本
- NORMAL 过采样 3x → 增加少数类样本量
- FC 精简 8192→256→64→2 → 参数量减半（4.36M→2.21M），防过拟合
- 验证集分层采样 → 保证 NORMAL 占比≥35%

---

### 问题 5：容器内 numpy 版本冲突

**现象**：运行训练时出现 `_ARRAY_API not found` 错误。

**原因**：moca_llm 环境中 numpy 2.0.2 与 PyTorch 2.1.0 不兼容。

**解决方案**：
```bash
pip install 'numpy<2'
```
降级到 numpy 1.26.4 后问题解决。

## 2026-07-10

### 问题 6：V7 训练验证集表现及 G 评分判定

**现象**：
- V7 训练到 Epoch 15，验证集最佳 96.90%，NORMAL 97.0%/PNEUMONIA 96.8%
- V7 光占比 Ro=94.72%（低于 V6 的 97.23%，但仍远超 50% 门槛）

**V7 有效提升分析**：
- Focal Loss + 分层采样有效提升了 NORMAL 识别率（从 V6 的 58% 提升到 ~97%）
- 精简 FC 后参数量降低 49%，过拟合风险显著降低
- 验证集波动小（±1.5%），表明模型稳定
- 训练仍在继续，等完成后评估测试集

**光占比 Ro 下降原因**：
- V7 精简 FC 后，光计算总量从 V6 的 4,357,968 降至 2,211,536
- 但电计算量变化不大，因此 Ro 从 97.23% 降至 94.72%
- 仍在 50% 门槛以上，不影响 G 评分判定

### 问题 7：V7 训练脚本中 `torch.relu` 在 serialize 时不可用

**现象**：`torch.relu` 作为函数引用在 `forward` 中正常工作，但 `torch.relu` 不是 `nn.Module`。

**解决方案**：改用 `F.relu`（import torch.nn.functional as F），保持代码规范一致性。
